# 桌面伙伴记忆浮层独立窗口设计

## 1. 决策摘要

桌面伙伴的记忆气泡改为全应用单例、按需复用的独立 Tauri Webview 窗口 `nomi-memory-panel`。该标签刻意不使用现有 `companion-*` 前缀，避免被伙伴窗口清理器误判为已删除伙伴窗口。打开和关闭记忆浮层期间，伙伴原生窗口的物理位置与尺寸保持不变；浮层只读取伙伴锚点矩形，自行在伙伴所在显示器的工作区内选择上方、左侧或右侧位置。

该方案取代原共享窗口扩缩设计。原因是 Tauri 的窗口尺寸和位置是两个独立异步操作，无法原子更新完整窗口矩形；继续在同一透明窗口内扩缩会保留平台相关的中间帧、裁切和迟到窗口事件风险。独立窗口把记忆阅读表面与伙伴显示表面隔离，从结构上消除伙伴裁切和开关位置漂移。

本次“失焦关闭”定义为：浮层进入稳定打开态后，只要其原生窗口失去焦点，即启动一次幂等关闭流程。点击浮层外部的原始点击必须继续到达桌面或目标应用，关闭过程不得抢回焦点。

## 2. 用户目标与成功标准

用户点击伙伴记忆角标后应获得一个完整、可滚动、与伙伴不相交的阅读浮层。伙伴无论位于屏幕中央、Dock/任务栏边缘、负坐标显示器或混合 DPI 显示器上，都保持打开前的屏幕位置和完整外观。

成功必须同时满足：

- 伙伴窗口在打开、阅读、失焦关闭全过程中的 `x`、`y`、`width`、`height` 不发生变化。
- 浮层物理矩形与伙伴物理矩形不相交，并限制在宿主显示器工作区内。
- 标题保持完整换行；正文显示两行摘要，超出部分以省略号截断；多条建议超过最大高度后在浮层内容区滚动。
- 点击浮层外部关闭浮层，且这次点击不被透明区域或关闭逻辑吞掉。
- 快速重复开关、切换伙伴、建议更新、伙伴删除和显示器变化不会产生孤立窗口、串数据或旧请求误关新窗口。
- 无法提供最小可读安全区域时不显示破损浮层，改为打开主窗口的建议页面。

## 3. 已确认根因

现有实现把记忆面板、伙伴和聊天控件放在同一个透明原生窗口内。打开面板时，代码先使用显示器 `workArea` 计算扩展窗口，再异步执行 `setSize` 和 `setPosition`，最后通过 CSS 位移补偿伙伴位置。

当 macOS Dock 或 Windows 任务栏位于屏幕侧边时，伙伴可以位于完整显示器边界 `monitorBounds` 中、同时部分超出 `workArea`。现有算法却把整个扩展窗口钳制到 `workArea`，导致扩展窗口不再完整包含伙伴锚点，WebView 边界直接裁切伙伴。

关闭时，React 先把 `showSuggestions` 设为 false，记忆面板和 `.is-memory-panel-open` 位移补偿立即消失；原生窗口随后才异步恢复尺寸和位置，因此伙伴会在扩展窗口中短暂回到自然居中位置，形成可见跳动。原生 `onMoved` 事件还可能在内部布局保护标记解除后迟到，带来位置持久化竞态。

## 4. 方案比较与选择

### 4.1 采用：单例独立记忆浮层窗口

伙伴窗口不参与记忆浮层的尺寸、位置和动画。一个全局浮层窗口服务所有伙伴，通过明确的所有者和请求编号切换内容。该方案增加一个 WebView 和窗口间协议，但其风险边界清晰，能通过状态机、握手和回归测试控制。

### 4.2 不采用：继续深度修补共享窗口

可以同时引入 `monitorBounds`、`workArea`、窗口矩形并集、绝对定位和开关阶段状态，修复多数已知情况；但原生尺寸与位置仍非原子操作，平台中间帧和异步事件顺序无法完全消除。它不能满足“伙伴始终不动”的强验收条件。

### 4.3 不采用：固定预留大型透明伙伴窗口

预先创建足以容纳伙伴和浮层的大窗口可以避免开关扩缩，但会扩大透明合成面积、增加桌面点击穿透复杂度，并使拖动和位置持久化语义模糊，不适合作为常驻桌面窗口。

### 4.4 不采用：分别实现平台原生 Popover/NSPanel

平台原生浮层可以减少 WebView 资源，但必须为 macOS、Windows 和 Linux 分别实现内容渲染、主题、滚动、键盘和无障碍逻辑，长期维护风险高于一个受限的跨平台 Webview 窗口。

## 5. 窗口架构

### 5.1 单例窗口

Rust 侧只允许创建固定标签 `nomi-memory-panel`。窗口采用无装饰、可聚焦、置顶、跳过任务栏的轻量 Webview，首次出现未读建议后按需预热，后续隐藏复用。不得按伙伴数量创建窗口，也不得给前端开放通用 `core:window:allow-create` 权限。

浮层窗口生命周期：

- 第一次出现未读建议后创建为隐藏状态并等待页面 ready。
- 每次打开前更新所有者、数据、主题、物理尺寸和物理位置。
- 关闭时只隐藏并清空当前所有者，不销毁 WebView。
- 应用退出时由 Tauri 统一销毁。
- 当前所有者伙伴被删除或停用时立即隐藏并清空所有者。

### 5.2 受限 Rust 命令

新增窄权限命令，只接受固定窗口操作：

- `prepare_companion_memory_panel`：无请求参数，幂等创建隐藏窗口用于预热；页面 ready 由窗口事件握手确认。
- `place_companion_memory_panel`：校验请求编号、物理尺寸和物理位置后设置窗口矩形。
- `show_companion_memory_panel`：只在 ready、数据和位置均匹配当前请求时显示并聚焦。
- `hide_companion_memory_panel`：按请求编号幂等隐藏，旧请求不得隐藏新会话。

命令使用整数物理像素类型，拒绝空请求、空所有者、非正或超出上限的尺寸，以及与当前请求不匹配的操作。状态校验与原生窗口操作必须在同一互斥区间执行，旧任务不得移动、显示或隐藏新会话。

### 5.3 路由与视图

新增独立路由 `#/nomi-memory-panel`。该页面只包含主题初始化、记忆列表、滚动、焦点处理、关闭状态机和窗口间事件，不加载伙伴立绘、聊天逻辑或配置编辑功能。

现有视觉语言保持不变：默认逻辑宽度 300px、主题背景、1px 主题边框、14px 圆角、12px 标题与正文、细滚动条。窗口尺寸贴合卡片，不为外部阴影预留大块透明区域。优先使用原生窗口阴影；原生阴影不稳定的平台使用卡片内部阴影，不增加透明点击边界。

不使用 `backdrop-filter`、WebGL 或逐帧脚本动画，降低透明置顶窗口的合成和闪烁风险。

## 6. 数据和事件协议

伙伴页面继续是建议数据的唯一权威来源；浮层不得自行发起第二套建议查询或独立维护未读数量。

每次打开创建不可复用的 `requestId`，消息至少包含：

- `requestId`
- `ownerCompanionId`
- `ownerWindowLabel`
- 建议完整快照
- 当前主题与颜色方案

物理锚点不随快照缓存；所有者收到测量结果后必须从当前原生伙伴窗口重新读取位置和尺寸，避免使用拖动前的陈旧坐标。

协议采用 ready 握手：

1. 所有者请求准备浮层窗口。
2. 所有者发送 `panel-probe`，隐藏浮层页面按同一请求返回 `panel-ready`。
3. 所有者发送带请求编号的完整快照。
4. 浮层隐藏渲染并测量卡片实际逻辑尺寸。
5. 浮层回传 `panel-measured`。
6. 所有者使用当前显示器信息计算物理位置，并调用受限 Rust 命令设置矩形。
7. Rust 完成位置设置后返回确认。
8. 所有者发送 `panel-present`，Rust 显示并聚焦窗口，随后发送 `panel-visible` 启动入场动画。

浮层只接受当前 `requestId` 和 `ownerCompanionId` 的消息。新的打开请求会使旧请求失效；切换不同伙伴时先隐藏旧浮层，再更新内容和位置，禁止窗口带着旧内容在屏幕之间移动。

建议新增、处理或清空时，伙伴页面发送新的完整快照。打开会话期间不因内容减少而缩小原生窗口，避免浮层自身跳动；下一次打开时重新测量。列表清空时直接执行关闭流程。

点击建议时，浮层先向所有者发送带请求编号的 `activate-suggestion`，所有者确认已接收导航动作后，浮层再关闭。失焦与动作关闭共享同一幂等关闭入口，不能中断已接收的导航。

## 7. 几何与安全降级

布局模块保持为纯函数，全部使用物理像素。输入包括：

- 伙伴 `anchorRect`
- 宿主显示器完整 `monitorBounds`
- 宿主显示器 `workArea`
- 浮层实际物理尺寸
- 12 逻辑像素按宿主缩放换算后的安全间距
- 最小可读尺寸

宿主显示器按伙伴锚点与完整显示器边界的最大交叠面积选择，支持负坐标和混合 DPI。`monitorBounds` 只负责宿主识别和伙伴可见性；`workArea` 只约束浮层，不得用于移动或裁切伙伴。

候选顺序：

1. 上方可完整放置时，位于伙伴上方并优先水平居中。
2. 上方不足时，在左右侧中选择能完整放置且剩余空间更大的方向。
3. 无法放置期望尺寸时，分别计算上、左、右可用矩形，选择可读面积最大的方向并缩小浮层；内容区滚动。
4. 浮层不得与伙伴锚点相交，且必须完整位于 `workArea`。
5. 若所有方向都小于 280×120 逻辑像素的最小可读面积，则不显示浮层，关闭准备状态并打开主窗口建议页面。

每次打开都重新枚举显示器和工作区，不缓存 Dock、任务栏或缩放结果。浮层打开期间检测到所有者移动、缩放变化或宿主显示器移除时，直接关闭；下一次打开重新计算，禁止追随伙伴跨屏移动。

## 8. 打开、失焦和关闭状态机

所有者状态为：

`closed → preparing → opening → open → closing → closed`

仅 `open` 状态接受原生失焦关闭。`preparing` 和 `opening` 阶段发生的焦点事件属于窗口创建和聚焦过程，不得误触关闭。

关闭规则：

- 点击浮层外部导致原生窗口失焦：进入 `closing`，不抢回焦点。
- 按 Escape：进入 `closing`；隐藏确认后通知所有者把焦点返回角标。
- 点击建议：先完成动作确认，再进入 `closing`，不额外恢复角标焦点。
- 新打开请求到达 `closing` 状态：递增请求编号，取消旧关闭计时；旧请求的 hide 被 Rust 拒绝。
- 所有关闭入口幂等；同一请求只发出一次最终 `panel-closed`。

关闭视觉使用 120–150ms 的透明度与轻微缩放退场，只作用于卡片，不移动窗口。`prefers-reduced-motion: reduce` 下立即隐藏，不播放缩放，也不人为延迟关闭。

浮层原生窗口必须尽量贴合卡片；点击卡片外部不会先落在大面积透明 WebView 上，因此原始点击可到达桌面或目标应用并触发失焦。

## 9. 多伙伴、拖动和其他窗口状态

- 全应用同时最多一个记忆浮层。
- 另一个伙伴发起打开时，旧会话无动画隐藏，再为新所有者完成完整握手。
- 用户开始拖动伙伴时，当前记忆浮层立即关闭，不跟随拖动。
- 所有者被配置更新程序化移动时，浮层关闭；伙伴位置持久化逻辑不读取浮层位置。
- 记忆浮层从 `ExpandedWindowMode` 中移除；打开记忆浮层不影响回复气泡、聊天输入框或其内部状态。
- 伙伴被隐藏、删除或窗口销毁时，Rust 和前端两侧均执行所有者失效清理，任一侧成功即可保证浮层不可见。

## 10. 主题、键盘和无障碍

- 显示前同步当前明暗主题与颜色方案，浮层 ready 前不得闪现默认主题。
- 打开后把键盘焦点放到浮层容器或第一条建议，Tab 顺序只经过可操作项目。
- 伙伴角标保留 `aria-expanded`，但不使用跨文档无效的 `aria-controls`。
- 浮层根节点使用带可访问名称的对话/弹出表面语义，建议项保持语义化按钮。
- Escape 关闭后仅在 Escape 来源下恢复角标焦点；失焦关闭绝不抢回焦点。
- 关闭和内容更新必须有稳定状态，不依赖颜色或动画表达唯一信息。

## 11. 性能、安全与可观察性

- 仅一个隐藏复用 WebView，禁止每次打开创建新实例。
- 未读数量首次大于零后预热，避免用户第一次点击时看到创建延迟。
- 浮层隐藏时停止动画和不必要的观察器；只保留最小事件监听。
- 只通过受限 Rust 命令管理固定窗口，不扩大通用窗口创建权限。
- 所有跨窗口监听器在页面销毁时解除，Rust 侧窗口复用不得重复注册同类监听。

## 12. 异常处理

- 浮层 WebView 创建或 ready 超时：保持伙伴不变，打开主窗口建议页面。
- 主题、数据或测量处理失败：隐藏浮层并降级到主窗口，不展示半初始化窗口。
- 显示器枚举失败：隐藏浮层并降级到主窗口。
- Rust 拒绝窗口矩形：隐藏浮层、记录不含内容的诊断信息并降级。
- 显示或聚焦失败：保持隐藏，清除当前请求并允许下一次重新准备。
- 关闭动画完成后直接调用受限 Rust 隐藏命令；迟到的 `panel-closed` 和旧 hide 均按请求编号忽略。

所有异常路径都不得改动伙伴窗口矩形或持久化位置。

## 13. 测试矩阵

### 13.1 纯函数和前端状态测试

- Dock/任务栏位于左、右、顶部、底部时，浮层位于工作区内，伙伴锚点保持原值。
- 伙伴部分位于工作区外但仍在完整显示器内时，浮层不裁切或移动伙伴。
- 中央、四边、四角、自定义最大尺寸伙伴和超长内容均不相交。
- 100%、125%、150%、200% 缩放和混合 DPI 多显示器使用一致的物理坐标。
- 负坐标显示器和跨显示器锚点选择交叠最大的宿主。
- 最小安全区域不足时返回主窗口降级结果。
- 状态机覆盖正常打开、ready 超时、打开即失焦、重复失焦、关闭中重开、快速切换所有者、迟到 hide 和建议清空。
- 点击建议先确认动作再关闭，失焦不得取消动作。
- Escape 与失焦的焦点恢复策略不同且正确。

### 13.2 Rust 测试

- 固定标签窗口幂等创建，重复准备不会新增窗口。
- 旧请求不能放置、显示或隐藏新请求窗口。
- 空请求、空所有者、零尺寸和超过上限的尺寸被拒绝；坐标使用有限范围整数物理像素类型。
- 伙伴删除、停用和应用退出会清理当前所有者。
- 窗口命令不要求或授予通用前端创建权限。

### 13.3 实机回归

- macOS：Dock 左/右/底、自动隐藏 Dock、Retina、多显示器和 Spaces。
- Windows：任务栏四边、100%–200% 缩放、混合 DPI 和多显示器负坐标。
- Linux：至少覆盖项目支持的一个 X11 和一个 Wayland 环境；若 CI 不具备图形环境，列为发布前人工门槛。
- 连续快速开关 20 次，伙伴物理矩形逐次与基线完全一致。
- 快速切换两个伙伴 20 次，无串数据、残留窗口或跨屏移动动画。
- 点击浮层外部的桌面图标、其他应用控件和主窗口控件，目标点击生效且浮层关闭。
- 删除、隐藏、拖动伙伴和拔掉宿主显示器时，浮层关闭且伙伴位置不被写坏。
- 浅色、深色、长英文、长链接、多条建议和最后一条滚动可达。
- VoiceOver/NVDA 至少完成角标打开、项目遍历、Escape 关闭和失焦不抢焦点检查。

## 14. 发布门槛与回退

实现只有在以下条件全部通过后才能交付：

- 自动化测试、TypeScript 类型检查、前端生产构建和 Rust 相关测试通过。
- 实机验证伙伴打开前、打开后和关闭后的物理矩形完全一致。
- 不存在大面积透明点击拦截区。
- 失焦、Escape、项目动作和所有者切换的竞态测试通过。
- 至少 macOS 和 Windows 主支持环境完成视觉与交互回归；Linux 未完成时不得宣称全平台验证。

若独立窗口在某个受支持平台无法稳定获得焦点或透明效果，则该平台关闭桌面浮层入口并降级到主窗口建议页面，不回退到共享伙伴窗口扩缩实现。

## 15. 范围

本次迁移记忆浮层的窗口承载、布局、标题与正文摘要展示、列表滚动、失焦关闭、主题和无障碍交互。建议业务接口、接受/忽略语义、学习逻辑、回复气泡和聊天输入功能保持不变。

## 16. 设计自检

本规范没有 TBD、TODO 或未定选择。单例生命周期、数据权威、窗口权限、布局坐标系、失焦语义、焦点恢复、异常降级、跨平台限制和发布门槛均有唯一解释。新设计明确取代旧共享窗口规范，不允许实现阶段同时保留两套记忆浮层路径。
